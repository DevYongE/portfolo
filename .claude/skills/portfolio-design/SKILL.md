# portfolio-design

포트폴리오 디자인 스펙 한 장(`_workspace/01_designer_spec.md`)을 만든다. 빌더가 이 한 파일만으로 구현 가능해야 한다.

## 절차

1. **사용자 정보 수집/확인** — 이름, 직무/정체성 한 줄, 기술 스택, 대표 프로젝트(2~5개), 연락처/링크(GitHub·이메일·LinkedIn), 톤 선호(미니멀/대담/전문적 등). 없는 항목은 가정하지 말고 `[채워주세요: ...]`로 남긴다.

2. **디자인 방향 결정** — `ui-ux-pro-max` 스킬을 호출해 스타일·색 팔레트·폰트 페어링·레이아웃을 고른다. 직접 발명하지 말고 그 라이브러리에서 포트폴리오 성격에 맞는 조합을 선택하라. 결정 결과를 hex·폰트명까지 구체적으로 적는다.

3. **섹션 구성** — 기본은 Hero / About / Projects / Skills / Contact. 채울 콘텐츠가 없는 섹션은 빼라. 각 섹션의 레이아웃을 텍스트로 기술(예: "Hero: 좌측 이름+한줄소개+CTA 버튼, 우측 프로필 이미지, 풀스크린 높이").

4. **카피 작성** — 각 섹션의 실제 문장 초안을 쓴다. 히어로는 정체성이 3초 안에 읽히게. About은 1~2문단. 프로젝트는 각각 제목/한줄설명/사용기술/링크.

5. **카피 휴머나이즈 (한글 카피일 때)** — 초안을 쓴 뒤 `humanize-korean` 스킬로 한글 카피를 다듬어 "AI가 쓴 티"를 제거한다. 단, 의미·사실은 절대 바꾸지 말 것(빈칸 `[채워주세요]`는 그대로 유지). 영문 카피엔 적용하지 않는다.

## 출력 형식 (`_workspace/01_designer_spec.md`)

```markdown
# 포트폴리오 디자인 스펙

## 디자인 방향
- 스타일: <스타일명>
- 팔레트: primary #xxxxxx / bg #xxxxxx / text #xxxxxx / accent #xxxxxx
- 폰트: heading <폰트> / body <폰트>
- 다크모드: 예/아니오

## 섹션 (순서대로)
### Hero
- 레이아웃: ...
- 카피: ...
### About
...

## 빈칸 (사용자 입력 필요)
- [채워주세요: ...]
```

## 원칙
- 측정 가능하게 적어라(hex·폰트명·섹션 순서). 리뷰어가 "스펙 대비 구현"을 대조할 수 있어야 한다.
- 빈칸을 placeholder 문장으로 메우지 마라. 명시적 `[채워주세요]`가 사용자에게 더 정직하다.
